# Session Menu Playbook (JTBD)

Документ описывает рабочие сценарии для `npm run session:menu` в формате Jobs to be Done.

## Логическая схема

```mermaid
flowchart LR
  A["Открыть проект"] --> B["Запустить session menu: npm run session:menu"]
  B --> C{"Что нужно сделать?"}
  C -->| "Начать новую задачу" | D["Start Session"]
  C -->| "Продолжить текущую ветку" | E["Sync Current Branch"]
  C -->| "Передать незавершенный прогресс" | F["Handoff Current Branch"]
  C -->| "Закончить задачу и открыть PR" | G["Finish Session"]
  C -->| "Слить ветку в main" | H["Merge Current Branch To Main"]
  C -->| "Проверить состояние" | I["Status"]
  D --> J["Запустить dev: npm run dev"]
  E --> J
  J --> K["Работа над кодом и контентом"]
  K --> C
  F --> L["Коллега продолжает работу в той же ветке"]
  G --> M["Ветка запушена + ссылка на PR"]
  H --> N["main обновлен и запушен"]
```

## Группа 1. Начать разработку в своей ветке

**JTBD**  
Когда я возвращаюсь к проекту и начинаю новую задачу, я хочу за 1 сценарий обновить `main` и создать рабочую ветку, чтобы сразу начать разработку без ручной git-рутины.

**User Story**  
Как разработчик, я хочу через `Start Session` выбрать тип ветки и имя, чтобы работать в корректной ветке от актуального `main`.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Start Session`
3. Выбрать тип ветки (`feat`, `fix`, `content`, `refactor`, `chore`, `docs`, `hotfix`)
4. Ввести короткое имя ветки
5. Запустить `npm run dev`
6. Начать изменения в файлах

## Группа 2. Продолжить работу и подтянуть изменения из main

**JTBD**  
Когда я уже работаю в ветке, я хочу безопасно синхронизироваться с актуальным `main`, чтобы не накопить конфликты к концу задачи.

**User Story**  
Как разработчик, я хочу одним действием сделать rebase моей ветки на `origin/main`, чтобы сохранить линейную историю и избежать поздних конфликтов.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Sync Current Branch`
3. Дождаться `fetch + rebase`
4. Продолжить работу (`npm run dev` может оставаться запущенным в другом терминале)

## Группа 3. Передать текущий прогресс коллеге (handoff)

**JTBD**  
Когда я не закончил задачу, но хочу передать её коллеге, я хочу сохранить WIP, запушить ветку и получить готовый handoff-пакет, чтобы другой разработчик сразу продолжил работу.

**User Story**  
Как разработчик, я хочу через `Handoff Current Branch` добавить commit message и заметку, чтобы передача контекста была быстрой и структурной.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Handoff Current Branch`
3. Подтвердить включение текущих изменений в commit (или пропустить)
4. Ввести сообщение коммита
5. Ввести handoff-note (опционально)
6. Получить на вывод:
- `branch`
- `branch_url`
- `pr_url`
- `note`
7. Отправить этот блок коллеге

## Группа 4. Завершить задачу и подготовить PR

**JTBD**  
Когда задача готова, я хочу пройти проверку и запушить результат с минимальным количеством ручных шагов, чтобы быстро открыть PR.

**User Story**  
Как разработчик, я хочу через `Finish Session` прогнать build, закоммитить изменения и получить ссылку на PR.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Finish Session`
3. Дождаться `npm run build`
4. При необходимости подтвердить commit и ввести commit message
5. Дождаться push ветки
6. Открыть ссылку compare/PR из вывода

## Группа 5. Слить ветку в main

**JTBD**  
Когда задача проверена и готова к релизу, я хочу одной командой выполнить merge ветки в `main`, чтобы зафиксировать результат в основной ветке.

**User Story**  
Как разработчик, я хочу через `Merge Current Branch To Main` обновить `main`, выполнить merge с подтверждением и запушить `main`.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Merge Current Branch To Main`
3. Подтвердить действие (`Continue? [y/N]`)
4. Скрипт делает:
- `fetch origin`
- `switch main`
- `pull --ff-only origin main`
- `merge --no-ff <current-branch>`
- `push origin main`

## Группа 6. Быстрая диагностика состояния

**JTBD**  
Когда я не уверен в текущем состоянии репозитория, я хочу быстро увидеть ветку, remote и изменения, чтобы принять корректное следующее действие.

**User Story**  
Как разработчик, я хочу запускать `Status`, чтобы за секунды понять контекст и не ошибиться с веткой/командой.

**Последовательность**
1. `npm run session:menu`
2. Выбрать `Status`
3. Проверить:
- текущую ветку
- remotes
- `git status --short`

## Минимальный рабочий цикл (рекомендовано)

1. `Start Session`
2. `npm run dev`
3. Работа над задачей
4. В середине работы периодически `Sync Current Branch`
5. Если передача коллеге: `Handoff Current Branch`
6. Если завершено: `Finish Session`
7. После ревью: `Merge Current Branch To Main`
